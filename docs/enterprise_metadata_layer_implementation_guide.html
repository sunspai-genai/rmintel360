1. Executive Overview

A governed enterprise data chatbot cannot be built safely by connecting an LLM directly to raw database tables. The LLM may understand the user question, but it does not inherently know which table, column, join path, metric definition, access policy, data quality rule, or business owner is correct.

The chatbot will be only as reliable as the metadata, metric registry, semantic layer, and governance controls behind it.

The enterprise implementation must create a metadata foundation that answers these questions:

  • Which tables and columns are approved for analytics?
  • What does each business term mean?
  • Which metrics are certified and how are they calculated?
  • Which dimensions can be used to group each metric?
  • Which joins are approved and what is their cardinality?
  • Which columns are restricted, confidential, PII, or masked?
  • What should the bot do when a question is unclear?
  • How will usage, generated SQL, feedback, and quality be audited?

The output of this work is not just documentation. It is a machine-readable semantic and governance layer that the chatbot uses at runtime.

2. Target Architecture

Client source databases and data lake/warehouse -> physical metadata scanner -> metadata catalog tables -> business glossary and metric registry -> semantic resolver and synonym index -> governance and access policy layer -> LLM orchestration -> SQL generation and validation -> controlled query execution -> answer, chart, citation, audit -> feedback and quality monitoring

7. Metric Registry Design

The metric registry is the most important table for SQL-generating chatbots. It tells the bot what it is allowed to calculate and exactly how.

Required Metric Fields

FieldDescriptionExample
metric_idStable unique ID.metric.average_deposit_ledger_balance
metric_nameUser-facing name.Average Deposit Ledger Balance
business_definitionApproved meaning.Average daily posted commercial deposit balance.
calculation_sqlFormula expression, not a full query.AVG(fdb.ledger_balance)
aggregation_typeSUM, AVG, COUNT, RATIO, CUSTOM.AVG
base_tableFact table used for metric.fact_deposit_balance_daily
required_columnsColumns needed by formula.ledger_balance, as_of_date, account_id
default_time_periodDefault when user omits time.latest_complete_month
valid_dimensionsAllowed groupings.customer_segment, product_segment, market, month
default_filtersApproved default filters.active_accounts_only=true
certification_statusDraft, certified, deprecated.certified
metric_ownerBusiness accountable owner.Deposit Analytics Owner
approved_byApprover.Finance Governance Council
versionMetric version.1.0

Metric Design Rules

  • Every SQL-generating metric must be certified before the chatbot uses it.
  • Do not let the LLM invent formulas for missing metrics.
  • Store formulas as expressions that can be inserted into governed SQL templates.
  • Define whether ratio metrics are weighted or unweighted.
  • Define valid time grain and default time filter.
  • Define valid dimensions for each metric to prevent invalid joins.

9. Table Grain and Join Path Catalog

SQL generation requires safe join logic. A chatbot must know how a metric base table can join to dimensions.

Join Path Fields

FieldDescriptionExample
join_path_idStable ID.join.deposit_balance_account
from_tableStarting table.fact_deposit_balance_daily
from_columnStarting join key.account_id
to_tableTarget table.dim_account
to_columnTarget join key.account_id
relationship_typeCardinality.many-to-one
join_typeINNER, LEFT, etc.INNER
certified_flagApproved join?true
descriptionBusiness explanation.Join daily balances to account attributes.

Rules

  • Every fact table must have a documented grain.
  • Every approved dimension must have a certified join path from the metric base table.
  • Do not allow fact-to-fact joins unless explicitly approved.
  • Document bridge tables and many-to-many joins carefully.
  • Document slowly changing dimension handling.

11. Data Quality Rules

A chatbot answer can look confident even when source data is stale or incomplete. Data quality metadata helps the bot decide whether to answer, warn, or block.

Rule typeExampleChatbot behavior
FreshnessBalance table must refresh by 7 AM daily.Warn or block if stale.
CompletenessAccount ID cannot be null.Warn if missing rate exceeds threshold.
ValidityRisk rating must be 1-10.Warn if invalid values exist.
ReconciliationTotal deposits must match GL within tolerance.Show certified status or caveat.
UniquenessCustomer ID unique in dim_customer.Protect join correctness.

Recommended Quality Rule Fields

quality_rule_id, table_name, column_name, rule_type, rule_sql, threshold, severity, owner, last_run_at, last_status, issue_url.

13. Semantic Search and Synonym Strategy

Users rarely type exact database names. The semantic layer needs synonym records and search documents.

Synonym Examples

User phraseTarget typeTarget IDRisk
balancemetricmultiple candidatesAmbiguous, ask clarification.
segmentdimensioncustomer_segment, product_segmentAmbiguous, ask clarification.
ledger balancemetricaverage_deposit_ledger_balanceCan resolve if exact.
channeldimensiontransaction_channelMay need subject context.

Best Practices

  • Keep synonyms curated and versioned.
  • Store confidence scores but do not let confidence override governance.
  • Use ambiguity groups for common terms like balance, segment, exposure, margin, income.
  • Log unresolved phrases and use them to improve metadata.

14. Clarification and Disambiguation Design

A production chatbot must ask follow-up questions when it cannot safely map user language to one governed meaning.

Clarification Triggers

  • Multiple metrics match the same phrase.
  • Multiple dimensions match the same phrase.
  • No certified metric exists.
  • Requested column is restricted.
  • Join path is missing or uncertified.
  • Time period is unclear and no default is approved.
  • User asks for a chart but axis mapping is unclear.

Clarification Response Pattern

I understood:
- Grouping: Product Segment

I still need:
- a certified governed metric

Choose one of these related certified metrics:
1. Loan Utilization Rate
2. Average Loan Outstanding Balance
3. Loan Commitment Amount

15. SQL Generation Guardrails

GuardrailRequired behavior
SELECT onlyBlock INSERT, UPDATE, DELETE, CREATE, DROP, COPY, CALL.
Allowed referencesParse SQL and reject tables/columns not in governed plan.
Restricted dataBlock restricted columns unless masking policy explicitly allows them.
Certified joinsOnly generate joins from metadata_join_path.
Query limitsApply row limits and internal limit + 1 for truncation detection.
TimeoutsSet warehouse/query timeout by role and workload.
Cost controlsBlock high-cardinality queries or require confirmation.
AuditPersist prompt, plan, SQL, validation, execution metadata.

17. Excel-to-Metadata Onboarding Process

Many clients begin with Excel data dictionaries. The implementation should standardize and load those files into metadata tables.

  1. Collect all existing Excel files, report definitions, wiki pages, ERDs, and BI semantic models.
  2. Map each column in the Excel files to the target metadata schema.
  3. Normalize names, data types, owners, and certification flags.
  4. Identify missing mandatory fields: grain, owner, definitions, PII, metric formulas, join keys.
  5. Load draft metadata into staging tables.
  6. Run validation checks and produce exception reports.
  7. Send exception reports to data stewards for cleanup.
  8. Promote approved records to production metadata tables.
Treat Excel as input, not the long-term metadata system of record. The chatbot needs governed metadata in database tables or an enterprise catalog API.

18. Metric Certification Workflow

Draft metric proposed -> data steward reviews definition -> data engineer validates SQL formula -> business owner approves meaning -> security reviews access and PII -> UAT validates sample answers -> metric marked certified -> chatbot can use it for SQL generation -> future changes create new version

22. Production Readiness Checklist

  • All in-scope tables have grain, owner, refresh, and certification status.
  • All certified metrics have formulas, owners, valid dimensions, and approvals.
  • All dimensions have physical mappings and security classification.
  • All joins used by chatbot are certified.
  • Restricted columns are blocked or masked.
  • SQL validator parses actual generated SQL references.
  • Query timeouts, limits, and cost controls are configured.
  • Cache keys include resolved metric and dimension plan.
  • New Conversation clears session state and cache as designed.
  • Audit logging captures prompt, semantic plan, SQL, validation, result metadata.
  • Golden question regression suite passes.
  • Feedback loop and support process are live.

24. Example End-to-End Onboarding Scenario

Subject Area: Commercial Deposits

  1. Identify core tables: deposit account, daily balance, deposit transaction, customer, product, branch, date.
  2. Document table grain: one row per account per date for daily balances.
  3. Define certified metrics: Average Ledger Balance, Average Collected Balance, Total Commercial Deposits, Transaction Amount.
  4. Define dimensions: Customer Segment, Product Segment, Market, Transaction Channel, Year Month.
  5. Certify joins from balance fact to account, customer, product, branch, date.
  6. Classify sensitive columns: customer name restricted, account number masked, tax ID blocked.
  7. Create synonyms: balance, ledger balance, collected deposits, segment, channel.
  8. Create golden questions and expected behavior.
  9. Run chatbot UAT with analysts and business users.
  10. Promote certified metadata and chatbot configuration to production.
