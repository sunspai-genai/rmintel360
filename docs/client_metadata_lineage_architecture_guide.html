Metadata and Lineage Layer Architecture Guide

A practical step-by-step implementation guide for designing the enterprise metadata, metric, semantic, governance, and lineage foundation required for a governed AI data assistant in a client environment.

1. Purpose

A governed AI data assistant is only as trustworthy as the metadata layer behind it. The assistant should never guess which table, column, metric, join, or lineage path to use. It should retrieve governed context, resolve the user's meaning against certified assets, ask clarification when there are multiple valid meanings, generate SQL only from approved definitions, and show source evidence.

The goal is not just to document data. The goal is to create a machine-usable metadata layer that can safely drive semantic resolution, SQL generation, source citation, lineage explanation, and governance audit.

What This Guide Helps You Build

  • A physical metadata inventory for enterprise tables and columns.
  • A business glossary with approved business definitions.
  • A certified metric registry with formulas, owners, and approved SQL expressions.
  • A governed dimension registry for attributes users can group, filter, and chart by.
  • A join path catalog so the assistant only joins tables using approved relationships.
  • A lineage model showing source systems, source objects, transformations, targets, owners, and refresh cadence.
  • A semantic and synonym layer so business language maps to governed data assets.
  • A curation and certification workflow for ongoing change management.

2. Architecture Principles

PrincipleMeaningWhy It Matters
Governed first The assistant may only use certified metadata assets for SQL and charts. Prevents hallucinated columns, unsafe joins, and inconsistent metrics.
Clarify before executing If a term maps to multiple assets, the assistant asks the user to choose. Prevents wrong answers for terms like balance, segment, revenue, exposure, or customer.
Separate business meaning from physical storage Business terms and metrics should not be buried only in table comments or BI reports. Lets non-technical users ask natural questions while SQL still uses physical assets safely.
Lineage is evidence Every trusted answer should trace back to source system, transformation, and target data asset. Builds confidence with business, audit, risk, and compliance teams.
Metadata is product data Metadata must have owners, quality checks, lifecycle, releases, and support SLAs. Keeps the assistant reliable after the MVP becomes production-grade.

5. Discovery Phase

Start with a structured discovery phase. Do not begin by connecting the assistant to every enterprise table. Begin with a narrow business domain, high-value questions, and known trusted datasets.

Information to Collect

CategoryQuestions to AskEvidence to Collect
Business use cases What questions do users ask repeatedly? Which decisions depend on these answers? Report catalog, dashboard screenshots, SQL extracts, business requirements.
Data platforms Where does the data live? Warehouse, lake, mart, ODS, CRM, core systems? Connection inventory, schemas, platform diagrams, access model.
Existing documentation Are table and column definitions in Excel, SharePoint, Collibra, Alation, Purview, dbt, or Confluence? Data dictionaries, glossary exports, metric documents, lineage documents.
Known pain points Which terms are ambiguous? Which metrics have multiple definitions? Examples: balance, exposure, revenue, segment, customer, active account.
Governance constraints Which columns are PII, PCI, confidential, restricted, or masked? Data classification rules, access policies, regulatory constraints.
Recommended starting scope: one business domain, 10-20 certified metrics, 10-15 certified dimensions, 5-10 core tables, and 30-50 golden test questions.

9. Business Glossary

The glossary defines business meaning. It should answer questions like "What does collected balance mean?" without generating SQL.

Glossary Fields

FieldPurpose
term_idStable unique identifier, such as term.collected_balance.
term_nameBusiness term shown to users.
definitionApproved business definition.
term_typeMetric, dimension, concept, policy, process, data quality rule.
calculationOptional plain-language calculation.
primary_tableOptional table mapping.
primary_columnOptional column mapping.
ownerBusiness owner accountable for definition.
certification_statusDraft, reviewed, certified, deprecated.
effective_dateWhen definition became valid.

Best Practices

  • Use business language first, physical mapping second.
  • Capture synonyms and forbidden synonyms.
  • Document common confusion points, such as ledger balance versus collected balance.
  • Version definitions when business meaning changes.
  • Require owner approval before a term becomes assistant-visible.

10. Metric Registry

The metric registry is the most important table for analytical questions. It tells the assistant which business measures are certified and exactly how to calculate them.

Metric Certification Rules

  • The formula must parse successfully with a SQL parser.
  • Every referenced table and column must exist in certified metadata.
  • The formula must not reference restricted columns unless explicitly allowed by policy.
  • The formula must have a business owner and technical owner.
  • The formula must pass reconciliation against an existing report or trusted SQL result.
  • Ambiguous metrics must have clarification options, not hidden defaults.
Never allow the assistant to invent metric formulas. Missing metrics should trigger a clarification or curation request, not generated SQL.

13. Lineage Capture

Use automated lineage capture where possible, but expect manual curation for business-friendly explanations. Automated lineage can identify technical flow; stewards must certify meaning.

Capture Methods

MethodWhat It CapturesBest ForLimitations
SQL parsing Table and column dependencies from SQL scripts, views, dbt models. Warehouse transformations. Dynamic SQL and stored procedures may need special handling.
ETL connector extraction Mappings from Informatica, DataStage, Talend, Airflow, Glue, ADF. Pipeline lineage. Tool APIs vary and may not expose all expressions cleanly.
BI scanning Semantic models, calculated fields, dashboards, report dependencies. Reconciling assistant answers with existing reports. Report calculations may duplicate or override warehouse metrics.
Manual curation Business explanations, source ownership, transformation notes. Client workshop outputs and critical metrics. Requires governance discipline and review.

Lineage Implementation Steps

  1. Identify critical metrics and columns requiring lineage.
  2. Capture source-to-target mappings from ETL, SQL, and BI artifacts.
  3. Normalize lineage into a standard table model.
  4. Attach plain-language transformation descriptions.
  5. Map lineage records to metric and dimension registry entries.
  6. Review lineage with data owners and certify.
  7. Expose lineage through the assistant as a metadata-only answer path.
  8. Monitor failed lineage questions and add missing lineage records.

15. Semantic Layer

The semantic layer maps user language to governed assets. It should support synonyms, ambiguity detection, domain context, and clarification options.

Semantic Assets

AssetExampleExpected Behavior
Synonym"ledger balance" -> Average Deposit Ledger BalanceResolve exact metric.
Ambiguous synonym"segment" -> Customer Segment, Product SegmentAsk clarification.
Aggregation signal"average", "total", "rate"Must match metric aggregation type.
Domain hint"loan", "deposit", "risk"Rank assets in the correct subject area.
Forbidden mapping"customer name" -> restrictedBlock SQL and offer safe alternatives.

Clarification Design

Clarification should feel guided, not like an error. The assistant should tell the user what it understood, what is missing, and which certified choices are available. 

Example:
User: Give me balance by segment.

Assistant:
I found more than one governed meaning.

Metric options:
1. Average Deposit Ledger Balance
2. Average Deposit Collected Balance
3. Average Loan Outstanding Balance

Grouping options:
1. Customer Segment
2. Product Segment

16. Governance Controls

Governance controls ensure the assistant is safe, explainable, and auditable.

Controls to Implement

ControlImplementationFailure Behavior
Certified asset only SQL generator accepts only certified metrics, dimensions, tables, joins. Ask clarification or reject request.
SQL AST validation Parse generated SQL and verify every table/column reference against approved plan. Reject SQL before execution.
Restricted column blocking Check PII and restricted flags before SQL generation and after SQL parsing. Block and offer safe dimensions.
Role-based visibility Use access policy metadata for SQL visibility, result visibility, export permissions. Mask, hide, or reject.
Audit packet Store resolved metric, dimensions, SQL, lineage, source citations, validation status. Do not show answer without audit evidence.

17. Data Quality

Data quality metadata allows the assistant to warn users when an answer is based on stale, incomplete, or low-quality data.

Recommended Quality Rule Fields

FieldDescription
rule_idStable quality rule ID.
asset_typeTable, column, metric, pipeline.
asset_nameAsset being checked.
rule_typeCompleteness, freshness, validity, uniqueness, reconciliation.
rule_sqlExecutable quality check SQL.
thresholdPass/fail threshold.
last_run_statusPassed, warning, failed, not run.
last_run_timeTimestamp of latest result.
severityLow, medium, high, critical.

Quality Rules to Start With

  • Freshness: table refreshed within expected SLA.
  • Completeness: required keys and measures are not null.
  • Uniqueness: dimension keys are unique where required.
  • Referential integrity: fact keys match dimension keys.
  • Reconciliation: certified metrics match source reports within tolerance.
  • Distribution drift: values are within expected ranges.

19. Metadata Ingestion

Metadata ingestion should be repeatable. Avoid one-time manual loading unless it is clearly marked as a temporary bootstrap step.

Recommended Pipeline

  1. Extract technical metadata from databases, catalogs, ETL tools, and BI tools.
  2. Load raw extracts into a metadata landing zone.
  3. Normalize source-specific fields into the standard metadata model.
  4. Run validation checks for missing owners, missing definitions, duplicate IDs, and invalid mappings.
  5. Stage records for steward review.
  6. Promote only approved records into certified metadata tables.
  7. Refresh the search/vector index used by the assistant.
  8. Create an audit record for every metadata change.

Excel Bootstrap Approach

If the client only has Excel data dictionaries, use them as an intake source, not the final system of record.

20. Curation Workflow

Curation is how metadata becomes trusted. The workflow should be explicit and auditable.

Recommended States

StateMeaningAssistant Visibility
DraftCreated but not reviewed.Hidden.
In ReviewPending steward/domain review.Hidden or admin only.
CertifiedApproved for assistant use.Visible.
DeprecatedStill exists but should not be used for new queries.Hidden or warning.
RejectedInvalid or duplicate.Hidden.

Approval Responsibilities

  • Business owner approves definition and usage.
  • Data owner approves source and lineage.
  • Data engineer approves physical mappings and joins.
  • Security owner approves classification and access policy.
  • AI/product owner approves assistant behavior and test results.

21. AI Assistant Integration

The assistant should use metadata through controlled services, not direct ad hoc table scans of the catalog.

Runtime Flow

  1. User asks a question.
  2. Assistant classifies intent: definition, metadata, lineage, analytical, chart, clarification, unsupported.
  3. Assistant retrieves relevant metadata context from search index.
  4. Semantic resolver maps user terms to certified metrics and dimensions.
  5. If ambiguous, assistant asks the user to choose from certified options.
  6. If analytical, SQL generator builds SQL from metric, dimensions, filters, and join catalog.
  7. SQL validator parses the generated SQL and rejects non-approved tables or columns.
  8. Query executor runs read-only SQL with limits and role controls.
  9. Answer generator summarizes result and attaches citations, SQL, chart, and governance audit.
  10. Feedback and failed-resolution events feed the metadata improvement backlog.

Assistant Must Never

  • Invent a metric formula that is not in the certified registry.
  • Join tables using guessed key names.
  • Use restricted columns unless policy explicitly allows it.
  • Answer ambiguous analytical questions without clarification.
  • Hide SQL, lineage, or source evidence from users who are allowed to see it.

22. Testing and Certification

Build a golden-question suite before go-live. Every certified metric and key dimension should have test prompts.

Test Types

Test TypeExamplePass Criteria
DefinitionWhat does average collected balance mean?No SQL, glossary answer, citation present.
AnalyticalShow average deposit ledger balance by customer segment.Correct metric, dimension, SQL, result.
ChartPlot loan utilization by month.Correct x-axis, y-axis, chart type.
ClarificationGive balance by segment.Asks for metric and dimension choice.
GovernanceList customer names with balance.Blocks restricted column.
LineageWhere does ledger balance come from?Returns source system, transformation, owner.
RegressionSame question repeated.Consistent result and cache behavior.

Certification Exit Criteria

  • All certified metrics reconcile to trusted reports or SQL.
  • All dimensions have approved table and column mappings.
  • All joins used by certified metrics are documented and approved.
  • All restricted columns are blocked or masked according to policy.
  • Golden-question pass rate meets agreed threshold, usually 95% or higher.
  • Failed questions have backlog items or explicit out-of-scope decisions.

25. Readiness Checklist

Before Connecting the Assistant to Client Data

  • Metadata Tables and columns harvested from client DB.
  • Glossary Business definitions reviewed by owners.
  • Metrics Priority metrics certified with formulas and reconciliation evidence.
  • Dimensions Grouping/filter attributes certified and classified.
  • Joins Approved join paths documented with cardinality.
  • Lineage Critical metric and column lineage captured and owner-approved.
  • Security PII/restricted columns classified and access policies loaded.
  • Quality Freshness, completeness, reconciliation, and uniqueness rules implemented.
  • Semantic Synonyms and ambiguity rules loaded.
  • Testing Golden-question suite passed.
  • Operations Feedback, issue triage, and curation workflow live.
The best production pattern is to launch domain by domain. Certify a small but high-quality metadata set, connect it to the assistant, monitor real questions, then expand coverage based on demand and risk.
