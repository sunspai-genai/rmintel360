1. Mental Model

Think of this application as a careful assistant sitting between a user and a banking database. The user asks a question in plain English. The bot decides whether the question needs a definition, SQL-backed analysis, or a chart. If SQL is needed, the bot must use only approved banking metadata.

The most important idea: the LLM can plan and write, but the governance layer decides what is allowed.

The app has five major responsibilities:

  1. Understand the user question.
  2. Retrieve governed metadata such as metrics, columns, tables, joins, and business terms.
  3. Generate SQL only from governed metadata.
  4. Execute safe SQL against the synthetic commercial banking database.
  5. Return a conversational answer, optional chart, SQL evidence, source citations, and audit details.

Plain-English Analogy

Application Part Analogy Why It Matters
Frontend The chat window Lets the user ask questions and see answers.
FastAPI backend The service desk Receives requests and sends responses.
LangGraph orchestrator The traffic controller Decides which step happens next.
LLM The analyst Interprets the question, writes SQL, plans charts, and summarizes results.
Governance validator The compliance officer Blocks unsafe or unapproved SQL.
DuckDB The local data warehouse Stores synthetic banking data and metadata.

4. End-to-End Request Flow

This is the main path when a user asks: Plot loan utilization by month.

1. Frontend
Sends the user message to /chat/message.
2. FastAPI
Receives the request, creates or continues a conversation, and calls the LangGraph orchestrator.
3. Cache Check
Looks for an existing answer for the same normalized question.
4. Metadata Retrieval
Searches governed metadata for relevant metrics, dimensions, tables, columns, synonyms, and terms.
5. LLM Planning
Classifies the intent and selects governed metric/dimension IDs.
6. SQL Generation
The LLM writes SQL from the governed query plan.
7. SQL Validation
The validator checks safety, access, approved columns, approved joins, and DuckDB compilation.
8. Execution
Runs validated SQL against DuckDB and returns a result table.
9. Answer + Chart
The LLM summarizes the result, the LLM plans chart axes, and Python Plotly builds the chart JSON.
10. Response
Returns answer, SQL, table, chart, citations, audit evidence, and feedback ID to the frontend.

Final Response Shape

The frontend receives a JSON object with fields like:

{
  "answer": "Loan Utilization Rate by Year Month returned 12 rows...",
  "generated_sql": "SELECT ...",
  "result_table": {
    "columns": ["year_month", "loan_utilization_rate"],
    "rows": [...]
  },
  "chart_spec": {
    "chart_type": "line",
    "x_axis": {"column": "year_month"},
    "y_axis": {"column": "loan_utilization_rate"}
  },
  "source_citations": [...],
  "llm_trace": {...}
}

5. Data and Metadata

The app uses a generated DuckDB file at data/commercial_banking.duckdb. It contains both banking data and governance metadata.

Synthetic Banking Tables

Table Purpose
dim_customer Customer attributes such as segment, region, industry, and restricted customer name.
dim_account Deposit account attributes and links to customers/products/branches.
dim_loan Loan attributes and links to customers/products/branches.
fact_deposit_balance_daily Daily deposit balances such as ledger balance and collected balance.
fact_deposit_transaction Deposit transactions by channel, type, and direction.
fact_loan_balance_monthly Monthly loan balances, commitments, and utilization metrics.
fact_relationship_profitability Customer relationship profitability measures.

Governance Metadata Tables

Metadata Table What It Stores Why It Matters
metadata_metric Certified metrics such as average deposit balance or loan utilization rate. The bot should calculate only certified metrics.
metadata_dimension Certified group-by attributes such as customer segment or year month. The bot uses these for grouping and chart axes.
metadata_column Physical columns, descriptions, data types, PII flags, and restricted flags. Blocks restricted columns such as customer name.
metadata_join_path Certified joins between tables. Prevents the bot from inventing joins.
metadata_business_term Business definitions and glossary terms. Answers definition-only questions without SQL.
metadata_synonym Alternative names users may type. Maps phrases like "segment" to governed dimensions.
metadata_lineage Source-to-target lineage records. Answers source and lineage questions.

Important Files

  • backend/app/synthetic_data/generate.py creates the DuckDB database.
  • backend/app/synthetic_data/metadata_seed.py creates metadata records.
  • backend/app/db/connection.py opens DuckDB connections.
  • backend/app/catalog/service.py reads metadata from the catalog.
  • backend/app/catalog/search_index.py builds and searches metadata documents.

Input and Output Example

Input to /chat/message:

{
  "message": "Plot loan utilization by month",
  "conversation_id": null,
  "user_role": "technical_user",
  "technical_mode": true,
  "limit": 100,
  "execute_sql": true
}

Output from /chat/message includes:

  • answer: the natural language answer.
  • generated_sql: SQL shown to technical users.
  • result_table: query rows and columns.
  • chart_spec: Plotly chart JSON if chart requested.
  • source_citations: metadata evidence.
  • audit_report: governance evidence.
  • turn_id: used for feedback.

7. LangGraph Orchestration

The main workflow is in backend/app/orchestration/llm_graph.py. LangGraph lets the app define a workflow made of nodes. In this project, the graph is simple and readable: 

memory -> retrieve -> interpret -> respond

Main Class

LlmGovernedAssistantGraph is the main orchestrator.

Function Input Output Role
invoke(...) User message, conversation ID, role, limit Final chat response dictionary Entry point used by the API.
_memory(state) Current state with conversation ID Recent turns and pending clarification Lets the bot remember follow-up context.
_retrieve(state) User message Metadata context and citations Finds governed metadata before the LLM decides.
_interpret(state) Message, memory, metadata context Intent decision or analytics result Calls the LLM semantic planner and applies policy.
_analytics(state, decision) LLM decision SQL, result, answer, optional chart Main SQL-backed analytics path.
_response(state) Final graph state Clean response for frontend/API Formats the final output.
The orchestrator is the best file to read first after api/chat.py, because it connects almost every other module.

8. LLM Layer

The LLM wrapper is in backend/app/llm/client.py. It hides the difference between real AWS Bedrock Titan and local fallback behavior.

Main Class: BedrockTitanClient

Function Input Output Role
invoke_json(...) Task name, system prompt, input payload, fallback function Python dictionary parsed from JSON Common entry point for all LLM calls.
_invoke_bedrock(...) Prompt and payload Raw model text Calls AWS Bedrock when enabled.
_parse_json(text) Model text Dictionary Extracts JSON from the LLM response.

Where the LLM Is Used

Intent and semantic planning SQL generation SQL repair Final answer summary Chart planning

Environment Settings

BANKING_ASSISTANT_BEDROCK_ENABLED=true
BANKING_ASSISTANT_AWS_REGION=us-east-1
BANKING_ASSISTANT_BEDROCK_MODEL_ID=amazon.titan-text-express-v1

If BANKING_ASSISTANT_BEDROCK_ENABLED=false, the app uses fallback functions so it can run locally.

9. Semantic Resolution

Semantic resolution means turning business language into governed objects. For example: 

"average deposit ledger balance by customer segment"

metric:    metric.average_deposit_ledger_balance
dimension: dimension.customer_segment

Two Layers Work Together

LLM Semantic Planner

In llm_graph.py, the LLM receives the user message and metadata context. It chooses intent, metric IDs, dimension IDs, chart need, and whether clarification is needed.

Governed Resolver

In semantic/resolver.py, the app resolves selected IDs into a governed query plan. It also catches ambiguity and restricted dimensions.

Important Outcomes

User Question Expected Outcome
What does average collected balance mean? Definition-only answer. No SQL. No chart.
Show average balance by segment May ask clarification because "average balance" can map to multiple metrics.
Plot loan utilization by month Chart query using loan utilization metric and year month dimension.
List customer names with average deposit balance Blocked because customer name is restricted.

10. SQL Generation and Governance

SQL generation is now LLM-first, but validation is mandatory. The SQL files are in backend/app/sql.

Files

File Role
sql/service.py Coordinates semantic resolution and SQL generation.
sql/llm_generator.py Asks the LLM to generate SQL and optionally repair failed SQL.
sql/generator.py Local governed fallback SQL template generator.
sql/validator.py Validates generated SQL before execution.

SQL Generation Flow

  1. The semantic resolver produces a governed_query_plan.
  2. llm_governed_sql_generator.generate(...) asks the LLM for DuckDB SQL.
  3. The validator parses the SQL and checks the actual referenced tables and columns.
  4. If invalid, the app gives the LLM one repair attempt.
  5. If still invalid, execution is blocked.

Validator Checks

Check Why It Exists
SELECT-only SQL Prevents updates, deletes, table creation, and unsafe operations.
Table allow-list SQL can only reference tables in the governed plan.
Column allow-list SQL can only reference approved columns in the governed plan.
Restricted column block Prevents use of columns such as dim_customer.customer_name.
Certified metric/dimension/join checks Ensures only governed objects are used.
DuckDB compile check Ensures SQL can actually run before execution.
Even if the LLM generates a query that looks correct, it is not executed unless sql/validator.py approves it.

11. Execution

Execution happens in backend/app/execution/service.py. The executor receives SQL that has already passed validation.

Main Class: GovernedQueryExecutor

Function Input Output Role
execute_from_message(...) User message and optional selected metric/dimensions QueryExecutionResult Generates SQL and executes it.
execute_sql_result(...) Already generated SQL result QueryExecutionResult Executes existing generated SQL without regenerating it.
_execute_sql(sql, limit) SQL text and row limit Result table dictionary Runs SQL in DuckDB and normalizes values.

Result Table Shape

{
  "columns": ["customer_segment", "average_deposit_ledger_balance"],
  "rows": [
    {"customer_segment": "Corporate Banking", "average_deposit_ledger_balance": 18855910.53}
  ],
  "row_count": 5,
  "limit": 100,
  "truncated": false
}

The executor fetches limit + 1 rows internally. If it gets an extra row, it knows the response was truncated.

12. Answers and Charts

Answer Generation

backend/app/answer/service.py creates the final natural language answer. The LLM receives the result table, SQL summary, governed query plan, and citations. It is instructed to summarize only facts present in the result.

Function Input Output
answer_from_message(...) User message Analytical answer after execution
answer_from_execution(...) Execution result Answer, key points, result overview
_llm_analytical_answer(...) Result table and governed plan LLM-written answer with fallback if needed

Chart Generation

backend/app/chart/service.py uses the LLM to plan the chart and Python Plotly to build the chart.

  1. The LLM chooses chart type, X axis, and Y axis.
  2. The app validates that selected columns exist in the result table.
  3. The app ensures the Y axis is numeric.
  4. Python Plotly builds chart JSON.
  5. The frontend renders the Plotly chart.
This is why the prior chart-axis flaws are guarded: the chart planner cannot use a missing column, and fallback logic prefers dimensions for X and metrics for Y.

13. Frontend

The frontend is intentionally simple: a professional chat interface for CFG Enterprise Data Bot. It lives in frontend/.

Main Files

File Role
frontend/index.html Page structure: header, messages area, and input form.
frontend/assets/styles.css Visual design: blue/green theme, chat bubbles, tables, feedback controls.
frontend/assets/app.js Browser logic: sends messages, renders answers, charts, SQL, sources, and feedback.
frontend/dev-server.mjs Local frontend server that proxies API calls to FastAPI.

Frontend Request Flow

  1. User types a message.
  2. sendChat(message) posts to /chat/message.
  3. The backend returns a JSON response.
  4. renderAssistantResponse(payload) renders the answer.
  5. If chart_spec exists, renderChart(...) calls Plotly.
  6. If result_table exists, renderTable(...) builds an HTML table.
  7. If generated_sql exists, renderDetails(...) shows SQL.
  8. If turn_id exists, feedback buttons are shown.

14. Feedback, Audit, Cache, and Curation

Feedback

Feedback is stored through backend/app/api/feedback.py and backend/app/feedback/service.py. The frontend sends thumbs up/down with the turn_id and conversation_id.

Governance Audit

backend/app/governance/audit.py builds an audit packet showing what metadata, SQL validation, source citations, joins, and policies were involved in the answer.

Cache

backend/app/cache/service.py caches complete assistant responses. Redis is used if enabled. Otherwise the app uses a local in-memory fallback. 

BANKING_ASSISTANT_REDIS_ENABLED=true
BANKING_ASSISTANT_REDIS_URL=redis://localhost:6379/0
BANKING_ASSISTANT_REDIS_CACHE_TTL_SECONDS=900

Admin Curation

backend/app/curation/service.py lets admins add or update governed business terms, metrics, dimensions, and synonyms. After curation, the app refreshes the search index and clears response cache.

Tests live in backend/app/tests. They are extremely useful for learning because each test shows an expected behavior.

Run All Tests

source .venv/bin/activate
python -m pytest backend/app/tests -q

Useful Test Files

Test File What It Teaches
test_orchestration_graph.py How chat requests move through the LangGraph flow.
test_sql_generation.py How SQL generation and validation should behave.
test_chart_generation.py How charts choose X and Y axes.
test_semantic_resolver.py How user phrases map to metrics and dimensions.
test_feedback_quality.py How feedback and quality monitoring work.

Regression Scenarios Already Covered

  • Plot loan utilization by month must group by year_month.
  • Bar charts by segment must use customer_segment on the X axis.
  • Follow-up clarification must preserve original dimensions such as "by channel".
  • Restricted columns such as customer name must not be used in generated SQL.
  • Definition questions must not generate SQL or charts.
  • Bad LLM SQL must be blocked by the validator.
  • Bad LLM chart axes must fall back to safe axes.
  • Repeated same questions should return consistent cached answers.

16. Suggested Learning Path

If you are new to coding, read the project in this order:

  1. Start with the frontend: read frontend/index.html, then frontend/assets/app.js. This helps you see what the user sends and what the UI expects back.
  2. Read the chat API: open backend/app/api/chat.py. This shows where the frontend request enters the backend.
  3. Read the orchestrator: open backend/app/orchestration/llm_graph.py. This is the main workflow.
  4. Read metadata retrieval: open backend/app/retrieval/metadata.py and backend/app/catalog/service.py.
  5. Read semantic resolution: open backend/app/semantic/resolver.py. This shows how business phrases become governed plans.
  6. Read SQL generation and validation: open backend/app/sql/llm_generator.py, backend/app/sql/service.py, and backend/app/sql/validator.py.
  7. Read execution and answer generation: open backend/app/execution/service.py, then backend/app/answer/service.py.
  8. Read chart generation: open backend/app/chart/service.py.
  9. Read tests last: tests show exactly what behavior the code must preserve.

Beginner Debugging Tips

  • When confused, search for the function name with rg "function_name".
  • Print or inspect the JSON response from /chat/message.
  • Use test files as examples of how functions are called.
  • Follow one question end-to-end before trying to understand every file.
  • Remember: request enters through api/chat.py and main logic happens in orchestration/llm_graph.py.
