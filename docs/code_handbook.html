CFG Enterprise Data Bot Code handbook

End-to-End Project Guide

This handbook explains how the governed commercial banking analytics assistant is wired: the frontend chat UI, FastAPI backend, LangGraph orchestration, LLM decision contract, metadata retrieval, SQL governance, DuckDB execution, charts, audit, exports, curation, feedback, and tests.

2. Architecture at a Glance

User asks a question in frontend/index.html.
frontend/assets/app.js posts the message to /chat/message.
FastAPI route backend/app/api/chat.py starts or resumes a conversation.
LangGraph flow backend/app/orchestration/llm_graph.py loads memory and retrieves governed metadata.
LLM client backend/app/llm/client.py asks Bedrock Titan for a strict JSON decision, or uses local fallback.
Backend policy enforces intent, response_mode, allow_sql, and allow_chart.
Only analytical/chart requests can generate SQL through backend/app/sql/service.py.
SQL is validated by backend/app/sql/validator.py before DuckDB execution.
Results are answered by backend/app/answer/service.py and optionally charted by backend/app/chart/service.py.
Frontend renders answer, table, chart, SQL, sources, and thumbs feedback.

3. Chat Request Flow

  1. Input: the browser captures a user message and sends JSON to POST /chat/message.
  2. Conversation: conversation_store.ensure_conversation creates or reuses a conversation id.
  3. Memory: the LLM graph loads recent turns to understand clarification replies.
  4. Retrieval: metadata_retriever.retrieve searches governed metadata and returns citations.
  5. LLM decision: llm_client.invoke_json returns structured intent and semantic choices.
  6. Policy gate: informational requests terminate before SQL; low-confidence requests ask clarification.
  7. Semantic resolution: analytics requests resolve to certified metrics, dimensions, and joins.
  8. SQL generation: the SQL generator builds SQL only from governed plans.
  9. SQL validation: sqlglot parses actual SQL references and checks allowed tables/columns.
  10. Execution: DuckDB runs validated read-only SQL.
  11. Answer: the answer generator summarizes the returned rows.
  12. Chart: chart generation runs only for chart_query.
  13. Audit: governance audit evidence is attached to the response.
  14. Persistence: the conversation turn is saved and a turn_id is returned.
  15. Feedback: thumbs up/down posts to POST /feedback.

4. LLM Intent Decision Contract

Bedrock Titan is used for language understanding, but backend code enforces the permission model. The LLM must return structured JSON with these fields. 

{
  "intent": "definition_question | metadata_question | lineage_question | table_discovery_question | analytical_query | chart_query | clarification_response | unsupported",
  "action": "information | analytics | clarify | unsupported",
  "response_mode": "definition_only | metadata_only | lineage_only | sql_answer | chart_answer | clarification | unsupported",
  "allow_sql": false,
  "allow_chart": false,
  "confidence": 0.94,
  "selected_metric_id": "metric.average_deposit_collected_balance",
  "selected_dimension_ids": ["dimension.customer_segment"],
  "chart_requested": false,
  "chart_type": null,
  "clarification_question": null,
  "clarification_options": [],
  "citations": []

8. Governance and Hallucination Controls

  • LLM is grounded: it receives retrieved metadata context and must choose only governed assets.
  • Intent policy is enforced: definitions, metadata, and lineage questions cannot run SQL or charts.
  • Public plans are not trusted: user-supplied governed query plans are ignored unless server-trusted.
  • Certified assets only: metrics, dimensions, and joins must be certified.
  • SQL is parsed: sqlglot extracts actual table/column references from generated SQL.
  • Restricted columns are blocked: metadata access controls are checked before execution.
  • Execution is read-only: SQL must be a single safe SELECT and compile in DuckDB.
  • Audit packet is attached: every chat response carries governance evidence.

10. Bedrock Configuration

Local fallback is used unless Bedrock is enabled. For AWS credential pairs or short-term session credentials, use standard AWS variables. For Bedrock short-term API keys, use AWS_BEARER_TOKEN_BEDROCK.

export BANKING_ASSISTANT_BEDROCK_ENABLED=true
export BANKING_ASSISTANT_AWS_REGION=us-east-1
export BANKING_ASSISTANT_BEDROCK_MODEL_ID=amazon.titan-text-express-v1

# For Bedrock API key authentication:
export AWS_BEARER_TOKEN_BEDROCK="paste-your-key"

With Bedrock enabled, the provider trace in chat responses should show aws_bedrock_titan. Without credentials, it should show a local fallback provider.

12. Common Flows to Trace

Definition only

User: What does average collected balance mean?
Expected: intent=definition_question, response_mode=definition_only, generated_sql=null, chart_spec=null

Ambiguous analytics

User: Give me average balance by segment.
Expected: needs_clarification with metric and dimension options

SQL-backed analytics

User: Show average deposit collected balance by customer segment.
Expected: SQL, table, answer, source citations, no chart

Chart

User: Plot loan utilization by month.
Expected: SQL, table, answer, chart_spec with line chart

Restricted data

User: List customer names with average deposit balance.
Expected: restricted identifier should be blocked or redirected to governed alternatives
