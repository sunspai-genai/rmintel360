1. Project Purpose

This MVP is a governed commercial banking analytics assistant. A user can ask plain English questions such as Plot loan utilization by month or What does average collected balance mean?. The bot decides whether the request needs metadata, SQL-backed analytics, a chart, or clarification.

The most important idea: the LLM can interpret user language, but the governance layer decides whether a metric, dimension, table, column, and SQL query are allowed.

The project supports:

  • Business definition answers from governed metadata.
  • SQL generation from certified metrics and dimensions.
  • Simple chart generation from SQL results.
  • Source citations and governance audit details.
  • Clarification questions with selectable choices when a request is ambiguous.
  • Session-aware cache reset for New Conversation.
  • Feedback capture and quality monitoring.

2. Beginner Mental Model

Think of the app as a team of small workers:

Frontend
Shows the chat UI, sends user messages to the backend, renders answers, tables, charts, SQL, citations, and feedback buttons.
FastAPI
Receives HTTP requests, routes them to services, and returns structured JSON responses.
LangGraph Orchestrator
Controls the conversation steps: memory, retrieval, interpretation, clarification, SQL, execution, answer, chart, cache.
LLM Client
Calls AWS Bedrock Titan when enabled; otherwise uses local deterministic fallback functions.
Catalog and Metadata
Stores governed tables, columns, metrics, dimensions, synonyms, lineage, and access policies.
Governance
Blocks unsafe SQL, restricted columns, unclear metrics, missing join paths, and hallucinated assets.

5. End-to-End Chat Flow

User sends message in browser -> frontend/assets/app.js sendChat() -> POST /chat/message -> backend/app/api/chat.py chat_message() -> conversation_store.ensure_conversation() -> llm_governed_assistant_graph.invoke() -> LangGraph nodes: memory retrieve interpret respond -> if informational: answer from metadata only -> if analytical/chart: resolve metric and dimensions ask clarification if ambiguous generate governed SQL validate SQL references and access controls execute SQL in DuckDB generate answer generate chart if requested build audit evidence -> record turn in conversation tables -> return ChatResponse JSON -> frontend renders answer, table, chart, SQL, sources, audit, feedback

The file to study first is backend/app/orchestration/llm_graph.py. It is the main brain of the chat experience.

6. LLM and Bedrock Flow

The LLM client lives in backend/app/llm/client.py. It has one main public function: invoke_json().

FunctionInputOutputWhat it does
BedrockTitanClient.invoke_json() Task name, system prompt, input payload, fallback function Parsed JSON dictionary Calls Bedrock Titan if enabled. If disabled or failing, calls local fallback.
_invoke_bedrock() Prompt and JSON payload Raw text from Bedrock Uses boto3 Bedrock runtime converse.
_parse_json() Raw model text Python dictionary Extracts and parses JSON even if model wraps it in markdown fences.
Even when Bedrock is enabled, LLM choices do not directly create SQL. The semantic resolver and SQL validator still apply governance checks.

7. Governance Layer

The governance layer prevents the bot from inventing or using unsafe data assets.

What is governed?

  • A metric is governed if it exists in metadata_metric with a certified definition and calculation.
  • A dimension is governed if it exists in metadata_dimension.
  • A table or column is governed if it exists in metadata_table or metadata_column.
  • A join is governed if it exists in metadata_join_path.

Governance checks

FilePurpose
semantic/resolver.pyResolves user words to certified metric and dimension IDs.
sql/validator.pyValidates SQL is SELECT-only and references only allowed tables/columns.
governance/audit.pyBuilds evidence for tables, lineage, access policy, and SQL validation.
catalog/service.pyReads metadata catalog records used by governance.

8. Clarification Engine

The app now uses a generic pending-task pattern for ambiguous analytical requests. This is important for questions like: can you give balance by segment.

Pending task structure

{
  "task_type": "analytical",
  "original_message": "can you give balance by segment",
  "intent": "analytical_query",
  "resolved_metric_id": null,
  "resolved_dimension_ids": [],
  "missing_slots": ["metric", "dimension"],
  "clarification_options": [...]
}

Important functions

FunctionInputOutputWhy it matters
_preflight_clarification() Current state and LLM decision Clarification response or None Runs semantic resolver before SQL so LLM-selected assets cannot bypass ambiguity.
_pending_choice_resolution() Current message and conversation memory LLM decision dictionary or None Checks whether this message is a follow-up to a pending clarification.
_resolve_pending_task_followup() Follow-up text and pending task Analytics or clarification decision Merges a user's follow-up selection into the original task.
_merge_pending_dimension_selection() Current dimensions, new dimensions, pending task Clean dimension ID list Prevents old and new dimension choices from being combined incorrectly.
_pending_task_clarification_question() Pending task Human-readable clarification text Explains what was understood, what is missing, and what choices are available.

Example

User: can you give balance by segment
Bot: I still need a certified governed metric and grouping.
     Metric options: Average Deposit Ledger Balance, Average Available Balance, ...
     Dimension options: Customer Segment, Product Segment

User: Product Segment
Bot: I already captured Product Segment as the grouping.
     I still need a certified governed metric.

User: Average Deposit Ledger Balance
Bot: Generates SQL grouped by product_segment only.

9. Cache and New Conversation

Cache logic lives in backend/app/cache/service.py. The current standard is:

  • Do not cache clarification responses.
  • Do not cache vague prompts as final answers.
  • Cache only after the request is resolved into a governed semantic plan.
  • Include resolved metric, dimensions, filters, chart settings, SQL, user role, limit, app version, model id, and metadata version in the cache key.
  • Clear cache when New Conversation is clicked.

Important functions

FunctionWhat it does
build_resolved_plan_key()Builds the safe cache key after metric/dimensions/SQL are known.
get()Checks Redis first, then local memory cache.
set()Stores cache with TTL in Redis and/or local memory.
clear()Clears local cache and Redis keys for both raw and resolved cache namespaces.

New Conversation reset

In the frontend, clicking New Conversation calls: 

POST /chat/session/reset

The endpoint is in backend/app/api/chat.py and calls assistant_response_cache.clear(). The frontend disables input while reset is running so a user cannot send a new question before reset finishes.

11. SQL Flow

semantic_resolver.resolve() -> governed_query_plan -> governed_sql_generator.generate() -> governed_sql_validator.validate() -> governed_query_executor.execute_sql_result()

Semantic resolver

backend/app/semantic/resolver.py converts natural language into a governed query plan.

FunctionRole
resolve()Main entry point. Returns resolved, needs clarification, information only, or unsupported.
_resolve_metric()Finds the certified governed metric.
_resolve_dimensions()Finds governed grouping dimensions.
_related_metric_options()Suggests related certified metrics when exact metric is missing.
_build_query_plan()Builds plan with metric, dimensions, base table, joins, filters.
_find_join_path()Finds certified join path from base table to target dimension table.
_restricted_dimension_ambiguities()Blocks restricted columns and suggests safe replacements.

SQL generator

backend/app/sql/generator.py builds SQL from the governed query plan. It does not decide governance. It receives already-approved metric and dimension objects.

SQL validator

backend/app/sql/validator.py protects execution. It checks:

  • Only SELECT statements.
  • Referenced tables and columns match the allowed governed plan.
  • Restricted columns are not used.
  • DuckDB can compile the query.

New Conversation behavior

When the user clicks New Conversation:

  1. The input is temporarily disabled.
  2. The frontend clears sessionStorage conversation id and recent conversations.
  3. The chat panel returns to the welcome state.
  4. The frontend calls POST /chat/session/reset.
  5. The backend clears local and Redis cache.
  6. The input is re-enabled.

Recent critical regressions covered

  • LLM-selected Product Segment cannot bypass clarification for can you give balance by segment.
  • Changing Product Segment to Customer Segment replaces the prior choice instead of grouping by both.
  • Clarification responses are not cached.
  • New Conversation clears backend cache.
  • Resolved cache keys do not mix different metric/dimension plans.

17. Debugging Checklist

If the bot skips clarification

  1. Check response fields: requires_clarification, clarification_options, pending_task.
  2. Check graph_trace. It should show preflight_clarification or clarification for ambiguous questions.
  3. Check semantic_result.ambiguities.
  4. Review _preflight_clarification() in llm_graph.py.

If cache seems wrong

  1. Click New Conversation or call POST /chat/session/reset.
  2. Check llm_trace.cache_status. A fresh ambiguous question should not be a cache hit.
  3. Verify final answers are cached by resolved plan, not vague raw prompt.
  4. Review build_resolved_plan_key() in cache/service.py.

If SQL uses wrong columns

  1. Inspect semantic_result.governed_query_plan.
  2. Check resolved metric ID and dimension IDs.
  3. Review generated SQL and sql_validation.
  4. Check metadata mappings in metadata_metric and metadata_dimension.

18. Glossary

TermMeaning in this project
Governed metricA certified business measure in metadata_metric with calculation, base table, required columns, and certification flag.
DimensionA business grouping attribute, such as Customer Segment, Product Segment, Market, or Year Month.
Pending taskConversation state that remembers an unresolved analytical request while the bot asks follow-up questions.
Resolved planThe complete governed semantic plan: metric, dimensions, joins, filters, assumptions.
ClarificationA guided follow-up question shown when the bot needs the user to choose a governed metric, dimension, or safe replacement.
LineageMetadata showing where data came from and how it was transformed.
Access policyGovernance record describing what roles can use which assets and whether masking applies.
Resolved-plan cacheSafe cache key built only after metric, dimensions, filters, chart settings, SQL, and metadata version are known.
