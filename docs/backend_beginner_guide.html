CFG Enterprise Data Bot Beginner backend guide

Backend Code Walkthrough

This guide is for a beginner who wants to understand the backend code step by step. It explains the project structure, how data moves through the application, and what the important modules, classes, and functions take as input and return as output.

Best way to use this guide: open it side-by-side with the code editor, start from backend/app/main.py, then follow the reading path below.

1. Simple Mental Model

The backend is a FastAPI application. Think of it as a set of layers:

API layer receives HTTP requests from the frontend.
Orchestration layer decides what type of question the user asked.
Retrieval and catalog layers fetch governed metadata.
LLM layer asks Bedrock Titan for structured decisions, or uses local fallback.
Semantic layer resolves metrics, dimensions, joins, and filters.
SQL layer generates and validates safe SQL.
Execution layer runs validated SQL against DuckDB.
Answer and chart layers create the final business response.
Conversation, audit, export, curation, and feedback layers persist supporting evidence.

2. Recommended Reading Order

  1. backend/app/main.py: learn how the app starts and which routers are registered.
  2. backend/app/core/config.py: learn where settings come from.
  3. backend/app/db/connection.py: learn how the app connects to DuckDB.
  4. backend/app/api/chat.py: learn the main chatbot endpoint.
  5. backend/app/orchestration/llm_graph.py: learn the main decision flow.
  6. backend/app/retrieval/metadata.py and backend/app/catalog/service.py: learn metadata retrieval.
  7. backend/app/semantic/resolver.py: learn how user language becomes governed metric/dimension IDs.
  8. backend/app/sql/service.py, generator.py, and validator.py: learn SQL generation and safety.
  9. backend/app/execution/service.py: learn how query results are returned.
  10. backend/app/answer/service.py and backend/app/chart/service.py: learn final response generation.
  11. Then read governance, export, curation, feedback, and synthetic data modules.

4. Main Chat Flow, End to End

Example user question:

Show average deposit collected balance by customer segment.
  1. Frontend sends the message to POST /chat/message.
  2. api/chat.py creates or reuses a conversation id.
  3. llm_graph.py loads recent conversation memory.
  4. metadata_retriever.retrieve() searches metadata for matching metrics, dimensions, columns, and lineage.
  5. llm_client.invoke_json() asks Bedrock Titan for a structured intent decision. If Bedrock is off, local fallback is used.
  6. _normalize_decision() and _policy_for_decision() make sure the decision is safe.
  7. semantic_resolver.resolve() resolves metric and dimensions.
  8. governed_sql_generator.generate() creates SQL.
  9. governed_sql_validator.validate() checks SQL safety and allowed columns.
  10. governed_query_executor.execute_from_message() runs the SQL.
  11. governed_answer_generator.answer_from_execution() summarizes results.
  12. governance_audit_service.build_from_chat_response() creates audit evidence.
  13. conversation_store.record_turn() saves the final turn and returns turn_id.

5. Input and Output Basics

What is a request?

A request is data sent into the backend. Example: ChatRequest contains the user message, role, conversation id, and options like execute_sql.

What is a response?

A response is data returned by the backend. Example: ChatResponse contains answer text, generated SQL, result table, chart spec, citations, audit report, and warnings.

What is a service?

A service is a class that performs business logic. Example: GovernedSqlService coordinates semantic resolution and SQL generation.

What is a schema?

A schema is a Pydantic model defining the shape of accepted input or returned output.

18. How to Debug as a Beginner

If the frontend says offline

Check backend is running at http://127.0.0.1:8000/health.

If no database exists

Run python3 -m backend.app.synthetic_data.generate.

If Bedrock is not used

Check BANKING_ASSISTANT_BEDROCK_ENABLED=true, region, model id, and credentials. Look at llm_trace.

If SQL is blocked

Look at sql_validation.errors, warnings, and audit_report.

source .venv/bin/activate
python3 -m compileall backend
node --check frontend/assets/app.js
python3 -m pytest backend/app/tests
