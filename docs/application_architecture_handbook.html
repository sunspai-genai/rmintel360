1. Application Overview

CFG Enterprise Data Bot is a governed commercial banking analytics assistant. It allows users to ask business questions, metadata questions, lineage questions, analytical questions, and chart requests through a chatbot interface.

The assistant does not simply translate text to SQL. It first retrieves governed metadata, classifies the intent, resolves business language to certified metrics and dimensions, asks clarification when terms are ambiguous, generates SQL only from governed metadata, validates the SQL, executes it, summarizes the result, and attaches citations and audit evidence.

The core design principle: the LLM may help interpret and phrase answers, but governance controls decide whether SQL, charts, and data access are allowed.

Supported Question Types

DefinitionExample: What does average collected balance mean?
MetadataExample: What columns are available in fact_loan_balance_monthly?
LineageExample: Where does ledger balance come from?
AnalyticalExample: Show average deposit ledger balance by customer segment.
ChartExample: Plot loan utilization by month.
ClarificationExample: Give me balance by segment.

3. Runtime Flow

This is the main path when a user sends a message from the chatbot.

User sends message. The browser calls POST /chat/message with message, conversation ID, user role, and execution settings.
Conversation is created or reused. conversation_store.ensure_conversation gives the turn a conversation context.
LangGraph starts. llm_governed_assistant_graph.invoke runs the assistant state machine.
Memory is loaded. The graph reads recent turns and pending clarification tasks.
Metadata is retrieved. The retriever searches governed metadata and builds source citations.
Intent and semantic decision are made. The LLM/fallback decides whether the request is informational, analytical, chart, clarification, or unsupported.
Governance policy is applied. SQL is blocked for definition, metadata, and lineage requests. Charts are allowed only for chart requests.
Clarification may be required. Ambiguous metric or dimension phrases return guided choices instead of SQL.
SQL is generated and validated. Analytical requests produce SQL only from certified metadata, then validate table and column references.
Query is executed. The executor runs read-only SQL with row limits and truncation handling.
Answer and chart are generated. The backend summarizes the result and optionally builds a Plotly chart spec.
Audit and response are returned. The response includes answer, table, SQL, chart, citations, governance audit, trace, and feedback ID.

  • Answer: main analyst-style response.
  • Answer Summary: metric, grouping, row count, governance status.
  • Result Table: returned data rows.
  • Chart: Plotly visualization when chart spec exists.
  • SQL: generated governed SQL when applicable.
  • Sources: expandable citations section.
  • Governance Audit: expandable governance evidence.
  • Feedback: thumbs up/down if backend returns a turn ID.

7. LangGraph Orchestration

The main chatbot flow lives in backend/app/orchestration/llm_graph.py. It uses LangGraph as a simple state machine with four main nodes.

NodeFunctionWhat It Does
memory_memoryLoads previous turns and pending clarification tasks.
retrieve_retrieveGets relevant metadata and source citations.
interpret_interpretUses LLM/fallback and backend policy to decide the path.
respond_respondReturns final state as the chat response.

Possible Routes

  • definition_only: answer from metadata, no SQL.
  • metadata_only: answer schema/table/catalog questions, no SQL.
  • lineage_only: answer lineage questions, no SQL.
  • sql_answer: generate SQL, execute, summarize.
  • chart_answer: generate SQL, execute, summarize, build chart.
  • clarification: ask user to choose governed metric/dimension.
  • unsupported: reject out-of-scope questions.

9. Intent and Semantic Resolution

Intent Classification

intent/classifier.py classifies the user's message. It identifies chart terms, lineage terms, metadata terms, definition terms, analytical terms, and unsupported requests.

Semantic Resolution

semantic/resolver.py maps natural language to governed assets. It resolves:

  • Metric: example metric.average_deposit_ledger_balance.
  • Dimensions: example dimension.customer_segment.
  • Join paths required to connect the metric base table to dimensions.
  • Default filters, such as last 12 months for month trends.
  • Ambiguities, such as balance or segment.
  • Restricted dimensions, such as customer name.
Recent hardening ensures exact metric names and aggregation words are respected. For example, "average loan outstanding balance" resolves to the average metric, not the total metric.

10. SQL Generation and Validation

SQL generation is split into generation and validation. This keeps the assistant from executing unsafe SQL.

ModuleRole
sql/service.pyCoordinates semantic resolution, SQL generation, and validation.
sql/generator.pyBuilds deterministic governed SQL from the query plan.
sql/llm_generator.pyLLM-assisted SQL generation path grounded in metadata.
sql/validator.pyValidates SQL references, policies, and compile safety.

Validator Responsibilities

  • Only allow read-only SELECT statements.
  • Parse SQL references and compare them against allowed tables and columns.
  • Block restricted or non-governed columns.
  • Compile SQL against DuckDB before execution.
  • Return clear validation errors if SQL is unsafe.

11. Execution and Answer Generation

ModuleInputOutput
execution/service.py Generated SQL result and user query. Rows, columns, row count, timing, truncation flag.
answer/service.py Execution result and original message. Answer text, key points, result overview.

The executor fetches bounded results and the answer generator turns those rows into a short analyst-style summary. The result overview is also used by the frontend to create the Answer Summary card.

12. Chart Generation

Chart generation is handled by chart/service.py. It takes an analytical answer result and creates a Plotly-compatible chart specification.

Chart Rules

  • Use a line chart for month/date trends.
  • Use a bar chart for categorical groupings.
  • Use the first dimension as the x-axis.
  • Use the metric as the y-axis.
  • Use the second dimension as series/color when two dimensions are present.
  • Reject invalid LLM chart axes and fall back to governed chart planning.

13. Governance and Audit

Governance exists throughout the application. It is not a separate afterthought.

governance/audit.py builds audit evidence from each chat response. The frontend renders this in an expandable Governance Audit section.

14. Conversation, Cache, Feedback

Conversation Memory

conversation/service.py stores conversation turns. This allows follow-up answers like "Customer Segment" or "Loan Utilization Rate" to complete a prior clarification request.

Cache

cache/service.py caches only final resolved responses. Cache keys include conversation, resolved metric, dimensions, filters, generated SQL, chart request, role, and execution settings. Ambiguous clarification prompts are not cached as final analytical answers.

New Conversation

The frontend clears browser session state and calls POST /chat/session/reset. The backend clears assistant response cache.

Feedback

feedback/service.py stores thumbs up/down, reason codes, comments, and quality summary. This is used to monitor weak answers and improve metadata.

16. LLM and Bedrock Integration

llm/client.py is the abstraction around AWS Bedrock Titan. If Bedrock is disabled or unavailable, the client uses a local fallback so the application still runs in development.

Where the LLM Helps

  • Intent and semantic decision support.
  • Grounded SQL planning/generation path.
  • Analytical answer wording.
  • Chart planning.

Where Deterministic Governance Still Controls

  • SQL permission by intent.
  • Certified metric and dimension requirement.
  • Restricted column blocking.
  • Join path approval.
  • SQL AST validation.
  • Cache scoping and final-response eligibility.

19. Beginner Walkthrough

If You Want to Understand the App Step by Step

  1. Start with frontend/index.html to see what the user sees.
  2. Open frontend/assets/app.js and find sendChat. This is where the UI calls the backend.
  3. Open backend/app/api/chat.py. This receives the chat message.
  4. Follow the call to llm_governed_assistant_graph.invoke in orchestration/llm_graph.py.
  5. Read the four graph nodes: _memory, _retrieve, _interpret, _respond.
  6. For definition questions, follow _information_answer.
  7. For analytical questions, follow _analytics.
  8. Inside analytics, read semantic/resolver.py to understand metric and dimension resolution.
  9. Then read sql/service.py, sql/generator.py, and sql/validator.py.
  10. Next read execution/service.py to see how SQL returns rows.
  11. Read answer/service.py and chart/service.py to see final response creation.
  12. Finally, read governance/audit.py to understand how evidence is attached.

Mental Model

User question
  -> Chat API
  -> LangGraph memory and metadata retrieval
  -> LLM/fallback intent decision
  -> Backend governance policy
  -> Semantic resolver
  -> SQL generator and validator
  -> Query executor
  -> Answer/chart/audit builder
  -> Frontend renders structured response
The safest way to learn this codebase is to pick one question, such as "Plot loan utilization by month", then trace it through frontend, chat API, orchestration, semantic resolver, SQL generation, execution, answer generation, chart generation, and final rendering.
